# Git для домашних работ  
Методические рекомендации для студентов

---

# 1. Зачем нам Git в курсе

Git позволяет:

- хранить историю изменений
- работать в отдельных ветках
- отправлять работу на проверку через Pull Request
- избегать перезаписи чужого кода

В курсе действует правило:

> Каждая домашняя работа выполняется в отдельной ветке.  
> Ветка main руками не редактируется.

---

# 2. Основные понятия (очень кратко)

**Репозиторий** — папка проекта с историей изменений.  
**Ветка (branch)** — параллельная версия проекта.  
**Коммит** — зафиксированное изменение.  
**Pull Request (PR)** — запрос на слияние вашей ветки в main.

---

# 3. Стандартный алгоритм сдачи домашней работы

## Шаг 0. Если это первый раз

```bash
git clone <ссылка-на-репозиторий>
cd <папка-репозитория>
```

---

## Шаг 1. Обновить main (ОБЯЗАТЕЛЬНО)

Перед каждой новой домашкой:

```bash
git checkout main
git pull
```

Почему это важно:  
Если не сделать `git pull`, новая ветка создастся от старой версии main,  
и при сдаче возникнут конфликты.

---

## Шаг 2. Создать новую ветку

```bash
git checkout -b hw_1
```

Теперь вы работаете в своей ветке.

Проверить текущую ветку:

```bash
git branch
```

---

## Шаг 3. Сделать изменения (написать код)

---

## Шаг 4. Проверить, что изменилось

```bash
git status
```

---

## Шаг 5. Добавить файлы в коммит

Добавить конкретную папку:

```bash
git add hw1
```

Или всё сразу:

```bash
git add .
```

---

## Шаг 6. Сделать коммит

```bash
git commit -m "Add HW1"
```

Сообщение должно быть коротким и понятным.

---

## Шаг 7. Отправить ветку на сервер

Первый раз:

```bash
git push -u origin hw_1
```

После этого достаточно просто:

```bash
git push
```

---

## Шаг 8. Создать Pull Request

- source branch: `hw_1`
- target branch: `main`
- добавить преподавателя как reviewer

После проверки ветка будет смёржена в main.

---

# 4. После того как PR смёржен

Перед следующей домашней:

```bash
git checkout main
git pull
git checkout -b hw_2
```

Это самый частый момент, где всё ломается.

---

# 5. Полезные команды для понимания ситуации

Посмотреть статус:

```bash
git status
```

Посмотреть историю:

```bash
git log --oneline --decorate --graph --all
```

Посмотреть ветки:

```bash
git branch
```

---

# 6. Если всё сломалось — делай вот это

Ниже — типичные ошибки и простые решения.

---

## ❌ Не даёт сделать push

Ошибка типа:

> failed to push some refs  
> non-fast-forward

Решение:

```bash
git pull --rebase
git push
```

---

## ❌ Git требует сначала закоммитить изменения

Ошибка типа:

> Please commit your changes or stash them before you merge

Решение:

```bash
git stash
git pull
git stash pop
```

---

## ❌ Локальный main не совпадает с сервером

Решение:

```bash
git checkout main
git pull
```

---

## ❌ Сделал(а) изменения не в той ветке

Если коммита ещё нет:

```bash
git checkout -b hw_branch
```

Изменения “переедут” в новую ветку.

---

## ❌ Добавил(а) лишние файлы через git add

Если коммита ещё нет:

```bash
git reset
```

---

## ❌ Нужно отменить последний коммит, но оставить код

```bash
git reset --soft HEAD~1
```

---

## ❌ Нужно полностью удалить последний коммит (ОСТОРОЖНО)

```bash
git reset --hard HEAD~1
```

---

# 7. Мини-чеклист перед созданием PR

1. Я в своей ветке?
2. Я сделал(а) commit?
3. Я сделал(а) push?
4. PR направлен в main?

---

# 8. Главное правило курса

> Новая домашка =  
> checkout main → pull → новая ветка

Если запомнить только это — 80% проблем исчезнут.


# Откаты в Git: что можно, что нельзя и когда

Git позволяет вернуться к прошлым версиям проекта.  
Но есть разные типы откатов — безопасные и опасные.

Главное различие:

- reset — переписывает историю
- revert — создаёт новый коммит, который отменяет старый

---

# 1. Если коммит ЕЩЁ НЕ запушен

В этом случае вы можете безопасно использовать reset.

## Сценарий: передумал(а) делать коммит

```bash
git reset --soft HEAD~1
```

Что произойдёт:
- последний коммит удалится
- изменения останутся в файлах
- можно снова сделать commit

Это безопасно.

---

## Сценарий: хочу полностью удалить последний коммит и изменения

```bash
git reset --hard HEAD~1
```

Что произойдёт:
- коммит исчезнет
- изменения исчезнут
- проект вернётся к предыдущему состоянию

⚠ Использовать осторожно — изменения будут потеряны.

---

# 2. Если коммит УЖЕ запушен

Вот здесь reset использовать НЕ рекомендуется.

Почему?
Потому что вы переписываете историю, и Git потребует:

```bash
git push --force
```

А force push:
- перезаписывает историю на сервере
- может удалить чужие коммиты
- опасен в командной работе

---

# 3. Правильный способ отката запушенного коммита

Используйте revert.

```bash
git revert <hash_коммита>
```

Что делает revert:
- создаёт новый коммит
- который отменяет изменения старого

История не ломается.
Force push не требуется.
Это безопасный способ работы в команде.

---

# 4. Если нужно вернуться в прошлое "посмотреть"

Можно перейти к старому коммиту:

```bash
git checkout <hash>
```

Но это режим detached HEAD —  
вы просто смотрите прошлое состояние.

Чтобы начать работу оттуда:

```bash
git checkout -b new_branch
```

---

# 5. Когда что использовать

| Ситуация | Что делать |
|----------|-----------|
| Коммит не запушен | reset |
| Коммит уже запушен | revert |
| Нужно просто исправить код | новый commit |
| Всё совсем запуталось | новая ветка от свежего main |

---

# Главное правило 

Если не уверены —  
НЕ используйте --force.

Лучше:
- сделать новый commit
- или использовать revert
- или начать новую ветку

Git почти всегда можно починить.
Потерянную историю — почти никогда.
